# 研究设计：基于知识图谱与检索增强生成的电影推荐系统

> 知识图谱和检索增强生成能否提升推荐性能与解释忠实度？

---

## 1. 研究动机

传统协同过滤（CF）推荐系统仅依赖用户-物品交互模式，存在两个公认的局限性：

1. **冷启动与长尾问题**：交互稀少的物品难以获得高质量推荐，因为 CF 缺乏足够的行为信号。
2. **缺乏可解释性**：CF 模型只能产生分数，无法以忠实于模型推理逻辑、且对用户有信息量的方式，说明*为什么*推荐某个物品。

知识图谱（KG）提供了结构化、人类可理解的关系（如共同演员、同一导演、类型重叠），有望同时缓解这两个问题。与此同时，检索增强生成（RAG）提供了一种基于检索文本证据生成自然语言解释的机制，有望提升解释忠实度，优于纯提示词驱动的 LLM 生成方式。

本项目研究将 KG 增强推荐与 RAG 解释生成相结合，是否能在推荐精度和解释质量两方面带来可度量的提升。

---

## 2. 研究问题与假设

| 编号 | 研究问题 | 假设 |
|------|---------|------|
| RQ1 | KG 增强的重排序能否在推荐精度上显著优于 CF 和内容基线？ | H1：在排序器中加入 KG 特征可显著提升 NDCG@K 和 Recall@K。 |
| RQ2 | KG 能否对长尾物品和兴趣集中的用户带来更大收益？ | H2：KG 特征对长尾物品 Recall 的提升幅度大于头部物品。 |
| RQ3 | RAG 生成的解释是否比纯提示词解释更忠实于检索证据？ | H3：RAG 解释在证据重叠度和语义一致性上优于纯提示词解释。 |
| RQ4 | 结构化知识（KG）与非结构化知识（RAG）在可解释推荐中是否互补？ | H4：结合 KG 和 RAG 的解释在覆盖率、多样性和用户信任度上优于单一方式。 |

**分阶段执行**：第一阶段聚焦 RQ1 + RQ2（KG 增强推荐），第二阶段聚焦 RQ3 + RQ4（RAG 解释生成）。

---

## 3. 系统架构

系统采用三阶段 **召回 → 重排序 → 解释** 流水线：

```
用户输入（历史行为、查询）
    |
    v
[阶段 1：召回] ---- Item-CF / BPR-MF / LightGCN
    |                 为每个用户生成 top-100 候选物品及分数
    v
[阶段 2：重排序] --- LightGBM 排序器
    |                 特征：CF 分数 + 内容相似度 + KG 特征
    |                 输出：重排序后的 top-K 列表
    v
[阶段 3：解释] ----- RAG 模块（第二阶段）
    |                 检索证据 → 提示 LLM → 生成解释
    v
最终输出：带解释的排序推荐列表
```

### 3.1 阶段 1：召回模型

三种基线召回模型，分别代表不同的范式：

| 模型 | 范式 | 方法 |
|------|------|------|
| **Item-CF** | 基于记忆的 CF | 基于用户交互向量的物品-物品余弦相似度；评分 = 用户历史物品相似度加权和 |
| **BPR-MF** | 基于模型的 CF | 通过 BPR 损失训练用户/物品嵌入；评分 = 用户与物品嵌入的点积 |
| **LightGCN** | 基于图的 CF | 在用户-物品二部图上进行多层图卷积；嵌入通过邻居聚合优化 |

三个模型均仅在训练集上训练，为每个用户输出 top-100 候选物品，并在相同测试集上用统一指标评估。

**多路召回融合**：最终采用 CF top-70 + KG top-50（TransE 嵌入余弦相似度）合并去重后取 top-100，覆盖从 1,105 部提升至 1,351 部电影。RotatE 嵌入用于排序阶段的特征计算（V3e/V4），不用于召回候选生成。

### 3.2 阶段 2：KG 增强重排序

LightGBM 排序器对阶段 1 的候选物品进行重排序。比较两种排序目标：

- **Pointwise**（二分类）：独立预测每个 (user, item) 对的 P(relevant)
- **LambdaMART**（列表级排序）：直接优化 per-user NDCG

**评估协议（Option C — 分布匹配训练）**：

一个关键设计选择是训练和评估均使用召回模型的 top-100 候选作为候选池。若使用随机负样本训练，会造成分布不匹配：随机负样本的 cf_score = 0（易于区分），导致模型学习捷径（如流行度偏差），在真实召回候选上失效。分布匹配训练确保排序器在训练阶段就能看到真实的困难负样本。

- **排序器训练**：召回 top-100 候选，用验证期交互标注（80% 用户）
- **排序器验证**：相同候选，剩余 20% 用户（用于早停）
- **排序器评估**：召回 top-100 候选，用测试期交互标注
- **特征计算**：所有特征均仅使用训练期数据，无信息泄露

**消融实验变体**：

| 变体 | 特征 | 目的 |
|------|------|------|
| **V1** | CF 分数 | 纯 CF 基线 |
| **V2** | CF 分数 + 内容相似度 + 流行度 + 评分数 | CF + 内容特征 |
| **V3** | V2 + 手工 KG 特征 | 完整 KG 增强模型 |
| **V3e** | V2 + KG 嵌入特征 | RotatE 嵌入特征 |
| **V4** | V3 + V3e（所有 KG 特征） | 全量 KG 特征 |

**KG 特征**（每个 (user, candidate_movie) 对，基于用户训练历史聚合）：

| 特征 | 描述 | 聚合方式 |
|------|------|---------|
| `shared_actor_count` | 共同演员数量 | sum, max |
| `shared_actor_idf` | IDF 加权的共同演员（稀有演员权重更高） | sum |
| `same_director` / `same_director_idf` | 是否共同导演（二值 + IDF 加权） | max |
| `same_genre_count` / `same_genre_idf` | 共同类型数量（计数 + IDF 加权） | sum, max |
| `co_liked_sum` | 候选与历史电影的协同共现次数（协同 KG 边） | sum |
| `same_decade_ratio` | 与候选同年代的历史电影比例 | ratio |

**IDF 加权**：共享稀有实体（如仅出演 2 部电影的小众演员）的权重高于共享常见实体（如出现在 50% 电影中的"剧情"类型）。IDF = log(N_movies / df_entity)。

**内容相似度**：候选电影元数据（类型 + 剧情简介）的 Sentence-Transformer 嵌入与用户历史电影嵌入均值的余弦相似度。

### 3.3 阶段 3：RAG 解释生成（第二阶段）

为每个推荐物品生成自然语言解释：

1. **查询构造**：用户历史（top-K 喜欢的电影）+ 候选电影
2. **双路检索**：稠密检索（Sentence-Transformer + FAISS）+ 稀疏检索（BM25），混合打分
3. **证据过滤**：去重、移除无关段落、控制 token 长度
4. **提示词设计**：结构化提示词，要求显式引用证据，禁止编造
5. **生成**：小型指令微调 LLM（如 Llama 2 7B / Qwen）

## 4. 数据集

### 4.1 用户交互数据：MovieLens 1M

| 属性 | 数值 |
|------|------|
| 用户数（过滤后） | 5,950 |
| 电影数（过滤后） | 3,125 |
| 正交互数（评分 >= 4） | 573,726 |
| 训练 / 验证 / 测试 | 398,867 / 54,680 / 120,179 |

**数据处理**：仅将评分 >= 4 视为正交互（隐式反馈）。按用户内时间顺序切分：训练（70%）/ 验证（10%）/ 测试（20%）。迭代过滤：用户 >= 10 次正交互，物品 >= 5 次正交互。

### 4.2 电影元数据：TMDB API

| 字段 | 描述 | 用途 |
|------|------|------|
| genres | 电影类型 | KG 构建、内容特征 |
| actors（前 5 名） | 主要演员 | KG 构建 |
| directors | 导演 | KG 构建 |
| overview | 剧情简介 | 内容相似度、RAG 证据 |
| keywords | 主题标签 | RAG 证据 |
| vote_average, vote_count | TMDB 社区评分 | 流行度特征 |

### 4.3 知识图谱

由 TMDB 元数据 + 用户协同交互模式 + 时间信息构建：

| 关系类型 | 数量 | 来源 | 描述 |
|---------|------|------|------|
| `co_liked` | 100,000 | 训练数据 | 被 >= 10 个用户共同喜欢的电影对（协同边） |
| `acted_by` | 18,044 | TMDB 演员表 | (Toy Story, acted_by, Tom Hanks) |
| `has_genre` | 8,871 | TMDB 类型 | (Toy Story, has_genre, Animation) |
| `directed_by` | 3,880 | TMDB 导演 | (Toy Story, directed_by, John Lasseter) |
| `released_in_decade` | 3,652 | TMDB 年份 | (Toy Story, released_in_decade, 1990s) |
| **合计** | **134,447** | | |

协同 `co_liked` 边将 CF 信号桥接到 KG 中，使得 RotatE 嵌入能同时捕捉结构化元数据和用户行为模式。

### 4.4 RAG 证据语料（第二阶段）

| 来源 | 内容 |
|------|------|
| TMDB overview | 每部电影的剧情简介 |
| TMDB tagline + keywords | 短描述性短语 |
| IMDb plot summary（可选） | 更详细的剧情描述 |

处理流程：句子级切分 → 去重 → Sentence-Transformer 编码 → FAISS 索引 + BM25 索引。

## 5. 评估

### 5.1 推荐指标（RQ1, RQ2）

所有指标均在召回模型的 top-100 候选上计算（Option C 评估）。

| 指标 | 定义 |
|------|------|
| **Hit@K** | 测试物品出现在 top-K 中的用户比例 |
| **NDCG@K** | 归一化折损累积增益，考虑排序位置 |
| **Recall@K** | 出现在 top-K 中的测试物品比例 |
| **MRR** | 第一个相关物品的排名倒数的均值 |
| **Coverage** | 出现在任意用户 top-K 中的物品占全部物品的比例 |

**统计显著性**：对 per-user NDCG@10 进行配对 t 检验（p < 0.05）。

### 5.2 长尾分析（RQ2）

- **定义**：训练集交互次数排名后 50% 的电影为"尾部"，其余为"头部"。
- **指标**：分别计算头部和尾部物品的 Recall@10。
- **用户兴趣熵**：按用户类型熵分桶，分析 KG 是否对兴趣集中的用户收益更大。

### 5.3 RAG 忠实度指标（RQ3，第二阶段）

| 指标 | 描述 |
|------|------|
| **证据重叠分数** | 生成解释与检索证据之间的 token/短语重叠度 |
| **语义相似度** | 解释与证据的嵌入余弦相似度 |
| **人工评判（1-5 分）** | 人工评估忠实度、信息量、相关性 |

**扰动实验**（4 个条件）：

| 条件 | 操作 | 预期结果 |
|------|------|---------|
| E1 | 原始证据 | 基线解释质量 |
| E2 | 移除关键证据句 | 解释质量应下降（验证对证据的依赖性） |
| E3 | 打乱证据顺序 | 轻微下降（验证鲁棒性） |
| E4 | 替换为无关证据 | 解释应发生显著变化（验证忠实度） |

### 5.4 结构化 vs. 文本分析（RQ4，第二阶段）

比较三种解释模式：

| 模式 | 知识来源 | 特点 |
|------|---------|------|
| **KG-only** | 图路径、共享实体 | 简洁、可追溯、逻辑清晰 |
| **RAG-only** | 检索文本段落 | 自然语言、更丰富 |
| **KG+RAG** | 两者结合 | 预期兼具可追溯性和丰富性 |

## 6. 技术栈

| 组件 | 技术方案 |
|------|---------|
| 深度学习 | PyTorch |
| 图神经网络 | PyTorch（自定义 LightGCN） |
| 排序器 | LightGBM |
| 知识图谱存储与查询 | NetworkX |
| 文本嵌入 | Sentence-Transformers (all-MiniLM-L6-v2) |
| 向量检索 | FAISS（第二阶段） |
| 稀疏检索 | BM25（第二阶段） |
| LLM 生成 | Llama 2 7B / Qwen（第二阶段） |
| 前端 | Streamlit |
| 评估 | scikit-learn, SciPy |
